#### F-008 — pgvector dimension claim is technically wrong
**File:** `docs/adr/0003-knowledge-graph-stack.md:53-54`

**Issue:** "Dimensions: 1024 (Voyage) or 1536 (OpenAI). Schema uses 1536 to accommodate either; smaller is fine with padding or migration." pgvector declares a fixed-dimension column — a `vector(1536)` cannot store a 1024-dim vector. "Padding" requires an application-layer pre-write step (and breaks cosine similarity unless padding is zero AND the index is rebuilt to assume so).

**Proposed correction:** Restate as: "Pick the embedding provider in M3.W2 and declare `embedding vector(D)` with D = chosen provider's dim (Voyage = 1024, OpenAI = 1536). Switching providers later requires a migration (re-embed all rows). Do not pad — pgvector dimension is fixed and similarity scores degrade if zero-padded."

- **Implementation cross-checks:** verified that the live build state agrees or disagrees with each ADR (`docs/adr/0002` vs PR #130 commit message produced F-004 and F-007). Did not run `pnpm` or inspect Docker images directly.
- **Self-review:** re-read this audit. No padding, no findings invented to look thorough. Three files declared clean. One section ("3 priority upgrades") flagged as missing rather than guessed at.
- **Time spent:** ~25 min reading, ~15 min cross-checking, ~20 min writing. Approx 1h total for 21 docs at this depth.
