## 3. Open bug — double-dispatch on daemon restart (root cause known, NOT fixed)

### Symptom

On daemon restart, **two architect subprocesses** start for the same task.

### Root cause

- `StateStore` (`~/.omc/ifleet/state.db`) persists sprint state across restarts.
- `Orchestrator` constructor calls `this.sprints.resumeAbandoned()`, which:
  1. Loads all sprints in `running` state.
  2. Resets their tasks from `running` / `assigned` → `pending`.
  3. Adds them to `activeSprintIds`.
- If the operator manually resets a unified-queue task to `pending` (via `sqlite3 state/tasks.db`) while a stale sprint for the same task still exists in `StateStore`, the tick loop creates a **new** sprint for that task.
- On the next tick, **both** the recovered old sprint and the new sprint call `dispatch` → two architects.

In normal operation (no manual DB intervention) this cannot happen, because the unified queue task stays `in_flight` while the old sprint is recovered, and `pickNext()` only returns `pending` tasks.

### Workaround used today

Before restart, cancel all running sprints in `StateStore`:

```sql
UPDATE sprints
   SET state_json = json_object(
        'kind', 'failed',
        'at', UNIXEPOCH() * 1000,
        'error', 'cancelled: stale sprint'
   )
 WHERE state_json LIKE '%"kind":"running"%';
```

### Proper fix (not implemented — next session)

Pick one:

- **(a)** When `recoverStale()` is called on `TaskStore`, also cancel orphaned sprints in `StateStore`.
- **(b)** `runTickLoop` should check whether a recovered sprint already covers the task before calling `submitSprint`.

- Worker agents **must** run in `.omc/worktrees/task-<id>` — never in the main repo. Never commit to `main` from a worktree agent.
- `SprintManager` emits events; it **never** calls GitHub directly. PRs that bypass the queue bridge must be refused. (CLAUDE.md rule.)
