## Smoke 3 — KG core populates

**Spec:** `docs/elevation/upgrades/03-knowledge-graph.md` + `src/agents/indexer/README.md`.

**Critical scope-clarification finding (not a regression):**

The smoke spec assumed that after M3.W1 merge, "a sprint runs → KG tables populate." This expectation is wrong against the actual M3.W1 deliverable. Per `src/agents/indexer/README.md`:

> | Done in this PR | Deferred |
> |---|---|
> | TypeScript + TSX parser (tree-sitter) | Python + Go parsers (M3.W5) |
> | Postgres schema (`deploy/postgres/0001-knowledge-graph.sql`) | GitHub webhook integration (M3.W3) |
> | Node + edge upsert (transactional, idempotent) | **Cross-repo link generation (M3.W4)** |
> | Voyage AI embedding pipeline (`voyage-code-3`) | **Architect runtime integration (M3.W3)** |
> | CLI: `pnpm graph:index <repoId> <pathToCheckout>` | Discord `/graph` commands (M3.W4) |
> | Architect tool stub (returns empty + log) | Reconciliation cron (M3.W5) |

And `src/agents/architect/tools/query_code_graph.ts:1-22`:

> M3.W1 scope: signature only. Returns an empty graph + logs `not yet integrated`
> so the architect can opt-in to the import path without behavior change. The
> actual ego-graph traversal + pgvector fallback land in M3.W3 alongside the
> planning-loop integration.

So: **a sprint run does NOT write to `code_nodes` / `code_edges` because no sprint code path invokes the indexer.** The indexer is a separate CLI (`pnpm graph:index <repoId> <path>`) that the operator runs explicitly, and the architect tool is a no-op stub.

**Environment check:**

- `.env.example` documents `IFLEET_KG_DATABASE_URL=postgresql://postgres:[YOUR-PASSWORD]@db.exswghbtgtdykklcsdxq.supabase.co:5432/postgres` and `IFLEET_KG_ENABLED=0` (default OFF).
- No `.env` file is present in this freshly-cloned worktree, so no credentials are configured here.
- No local Postgres on this workstation.

**What CAN be verified offline:**

1. **Schema migration file present and well-formed:** `deploy/postgres/0001-knowledge-graph.sql` defines 4 tables — `code_nodes` (with `vector(1536)` embedding column), `code_edges`, `cross_repo_links`, `indexer_errors` — plus HNSW index on `code_nodes.embedding`, uniqueness constraints, and check constraints on `kind`/`stage`/`link_kind`. All `CREATE` statements use `IF NOT EXISTS` — idempotent re-run is safe.
2. **Indexer unit + integration tests pass:** `pnpm vitest run src/agents/indexer` → **3 files, 16 tests pass** in 4.03s. This includes parser tests and upsert idempotency.
3. **Architect stub returns the expected empty result without crashing** (covered by `src/agents/architect/tools/__tests__/query_code_graph.test.ts`).

**Tables-touched-by-a-sprint count: 0.** This is by design.

**Verdict: SCHEMA PASS / RUNTIME DEFERRED.** The KG layer is correctly scaffolded for M3.W3 to land on top of. The smoke spec's "kg_* tables receive rows when a sprint runs" expectation will become testable once M3.W3 wires the architect tool to the real query path AND M3.W3 / M3.W5 introduces the webhook indexer that ingests on commit. Until then, the only way to populate the KG is the operator running `pnpm graph:index` against a checkout — which is out of scope for an end-to-end sprint smoke.

**Schema field corrections to T5's brief:** the actual table names are `code_nodes`, `code_edges`, `cross_repo_links`, `indexer_errors` — NOT `kg_nodes` / `kg_edges` / `kg_chunks` as the brief speculated. The brief's `kg_chunks` table does not exist in the M3.W1 schema (the embedding lives on `code_nodes.embedding`, not a separate chunks table).
